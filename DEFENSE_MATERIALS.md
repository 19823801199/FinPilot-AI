## 2. 一分钟介绍

FinPilot AI 是一个面向金融学习与研究的 AI 原生一体化平台。它将大语言模型、Multi-Agent 协作体系和 RAG 增强知识库深度融合，为用户提供三大核心能力：**智能学习** — 通过 AI 驱动的课程体系帮助用户掌握金融知识；**金融研究** — 多 Agent 专家协同完成股票分析、财报解读等复杂任务；**知识管理** — 支持私有文档上传与智能问答，让用户构建个人金融知识库。

与传统金融工具相比，FinPilot AI 打破了过去信息获取、学习与实操之间的壁垒。用户无需在多个软件之间切换，一个平台即可完成从学习到研究的完整工作流。我们使用 DeepSeek 大模型、ChromaDB 向量数据库、SSE 实时流式传输等前沿技术，确保回答专业、准确、实时。

FinPilot AI 的定位不是替代金融分析师，而是成为每一个金融从业者和学习者的 AI 副驾驶，让金融工作更高效、更智能。

## 5. 创新点

### 1. AI Native 全栈设计
FinPilot AI 从底层架构到上层交互，全部围绕 AI 能力进行设计。前端采用流式渲染适配 AI 输出的实时性，后端 Agent 编排层原生支持 AI 任务调度，每一个功能模块都深度集成了 AI 能力，而非简单地在传统应用中套一层 AI 外壳。

### 2. Multi-Agent 协作体系
我们设计了一套专家 Agent 协作体系，多个专业 Agent（学习专家、研究分析师、知识专家、报告撰写专家）协同完成复杂金融任务。Orchestrator Agent 负责任务分解、调度和结果聚合，实现了"1+1>2"的效果，大幅提升了复杂金融分析任务的质量和完整性。

### 3. RAG 增强知识库
私有文档上传后经过 Embedding 向量化存储到 ChromaDB，用户提问时系统先进行语义检索，找到最相关的文档片段，再结合 LLM 生成精准回答。这使得 AI 能访问和处理用户的私有数据，同时保证数据不出域，兼顾了智能性和安全性。

### 4. 流式交互体验
基于 SSE 协议实现了 AI 输出的实时流式推送。用户在提问后几乎立即看到 AI 开始回复，字符逐字呈现，大幅降低了等待焦虑，提升了交互的自然感和实时性。

### 5. 一体化金融学习平台
将"学习—研究—知识管理"三个原本分离的场景整合到一个平台，用户无需在多个工具间切换。从学习金融知识，到开展股票研究，再到管理个人知识资产，形成完整闭环。

## 7. 为什么使用 Multi-Agent

### 问题背景
金融分析是一个高度复杂的任务，涉及多个专业领域：公司基本面分析、财务数据解读、技术指标分析、行业趋势研判、风险评估等。单一 AI Agent 难以在所有领域都表现出色。

### Multi-Agent 的设计逻辑

我们类比人类团队的工作方式：一个完整的金融分析团队通常包含基本面分析师、技术分析师、风险经理和报告撰写人。FinPilot 的 Multi-Agent 体系正是模拟了这种协作模式：

- **Orchestrator Agent（协调者）** — 理解用户意图，将复杂任务分解，协调各 Agent 工作，聚合最终结果
- **Financial Analyst Agent（基本面分析师）** — 专注财务数据、估值分析、公司基本面评估
- **Technical Analyst Agent（技术分析师）** — 专注价格走势、技术指标、图表模式分析
- **Report Writer Agent（报告撰写专家）** — 将多源分析结果整合为结构化的研究报告
- **Learning Expert Agent（学习专家）** — 负责课程讲解、知识点拆解、练习生成

### Multi-Agent vs 单 Agent 的对比

| 维度 | 单 Agent | Multi-Agent |
|------|---------|-------------|
| 专业性 | 一个 Agent 覆盖所有领域，深度不够 | 每个 Agent 专注一个领域，专业性强 |
| 可维护性 | Agent Prompt 极其复杂，难以调试 | 每个 Agent Prompt 简洁清晰 |
| 可扩展性 | 新增能力需要重训练或重构 | 新增能力只需注册新 Agent |
| 结果质量 | 结果泛泛，容易遗漏关键维度 | 多维度分析，结果全面深入 |
| 计算效率 | 单次大模型调用，Token 消耗大 | 并行执行，整体效率更高 |

### 结论
在金融分析这种需要多维度专业知识的场景中，Multi-Agent 架构是显著优于单 Agent 的选择。它让 AI 系统具备了"团队协作"的能力，能够产出更专业、更全面的分析结果。

## 8. 为什么使用 RAG

### 问题背景
金融领域对信息的时效性和准确性要求极高。LLM 的训练数据存在截止日期，无法涵盖最新的市场动态、公司财报、政策变化。同时，金融从业者拥有大量私有文档 —— 内部研究报告、客户资料、历史交易记录等，这些数据不能上传到公有 AI 服务。

### RAG 的核心价值

**1. 突破知识边界**
LLM 的知识截止于训练数据的时间点。通过 RAG，我们可以将最新的金融资讯、公司公告、研报等实时纳入知识库，让 AI 的回答始终保持时效性。

**2. 隐私与安全**
用户的私有文档不出本地（或私有部署环境），文档内容经过向量化处理后仅以数值形式存储，无法反向还原原文。结合权限控制，确保数据安全。

**3. 可溯源性**
RAG 机制允许 AI 在回答时引用具体的文档来源。用户可以追溯到原始文档，验证信息的准确性，这在金融合规场景中至关重要。

**4. 领域针对性**
通用 LLM 在金融领域的专业知识存在盲区。RAG 可以将专业的金融文档（如会计准则、监管文件、分析方法论）注入到 AI 的推理过程中，显著提升回答的专业性。

### 我们的 RAG 实现

```
用户提问 → 问题 Embedding → ChromaDB 向量检索 → 召回 Top-K 相关片段
→ 拼接 Prompt（原始问题 + 检索结果） → DeepSeek 生成回答 → SSE 流式输出
```

- **向量数据库**：ChromaDB，轻量级、高性能、易于部署
- **Embedding 模型**：高效文本向量化，支持语义检索
- **切片策略**：智能文档分块（Chunking），保持语义完整性
- **检索策略**：支持相似度阈值过滤，确保召回质量

### 结论
在金融场景下，纯 LLM 无法满足时效性、隐私性和专业性的三重需求。RAG 是解决这些问题的最优技术方案，也是 FinPilot 知识库模块的核心技术支撑。

## 10. 评委可能提问30个问题及标准回答

### 技术架构类

**Q1: 你们的整体技术架构是怎样的？**
A: FinPilot 采用 Monorepo 架构（pnpm workspace），分为前端（React + TypeScript）、后端（Node.js + Express）、AI 三层。AI 层包含 Agent 编排引擎（Orchestrator Pattern）、RAG 引擎（ChromaDB + Embedding）和流式输出引擎（SSE）。所有模块通过统一的 API 网关通信，前后端通过 RESTful API 交互，AI 层通过 SSE 实现实时推送。

**Q2: 为什么选择 Monorepo 架构？**
A: Monorepo 方便代码共享和类型统一。前后端共用 TypeScript 类型定义，减少了接口对接成本。pnpm workspace 提供了高效的依赖管理和构建缓存，提升了开发效率。对于 FinPilot 这种前后端紧密耦合的项目，Monorepo 是最高效的选择。

**Q3: 前端用什么技术栈？为什么？**
A: 前端使用 React + TypeScript，搭配 Vite 构建工具。React 的组件化开发方式适合构建复杂的交互页面，TypeScript 提供类型安全，Vite 提供快速的开发体验。此外，React 的流式渲染能力很好地适配了 SSE 实时输出场景。

**Q4: 后端为什么选择 Node.js 而不是 Python？**
A: 选择 Node.js 主要基于两个原因：一是与前端共享 TypeScript 类型，减少上下文切换成本；二是 Node.js 的事件驱动模型天然适合 SSE 流式输出场景。在 Monorepo 架构下，前后端使用同一语言大幅提升了开发效率。AI 推理层仍然可以通过调用 Python 服务来实现，两者并不冲突。

**Q5: 你们的 API 设计遵循什么规范？**
A: 遵循 RESTful API 设计规范。所有 API 端点使用语义化的 URL 路径，标准的 HTTP 方法（GET/POST/PUT/DELETE），统一的请求/响应格式。流式接口使用 SSE 协议，提供了更好的实时体验。

### AI 决策类

**Q6: 为什么选择 DeepSeek 而不是 GPT 或其他模型？**
A: DeepSeek 在中文金融领域的表现出色，且 API 成本远低于 GPT-4。同时 DeepSeek 支持开源部署，未来可以实现私有化部署，满足金融客户的数据合规需求。在同等参数规模下，DeepSeek 的推理速度也更具优势。

**Q7: 如果 DeepSeek API 出现故障，你们的容错机制是什么？**
A: 我们设计了模型路由层，支持配置多个模型供应商作为备用。当主模型不可用时，自动切换至备用模型（如通义千问、文心一言等）。同时，RAG 知识库中的缓存策略可以部分缓解 API 不可用带来的影响。

**Q8: 你们如何处理大模型输出的"幻觉"问题？**
A: 主要采用三种策略：（1）RAG 约束 — 通过检索真实文档来约束生成内容，减少凭空编造；（2）Prompt 工程 — 在 System Prompt 中明确要求 AI 在不确定时声明"无法确认"而非猜测；（3）用户提示 — 在 UI 中标注"AI 生成内容仅供参考"，引导用户理性看待 AI 输出。

**Q9: DeepSeek 的上下文长度是否能满足你们的场景？**
A: DeepSeek 支持 8K-32K 的上下文窗口，对绝大多数字号场景已经足够。对于需要处理超长文档的场景（如数百页的年报），我们通过 RAG 切片策略，只将最相关的片段送入上下文，有效规避了上下文长度限制。

**Q10: 你们有没有对模型进行微调（Fine-tuning）？**
A: 目前我们主要采用 Prompt Engineering + RAG 的方式，没有进行模型微调。原因有二：一是微调的维护成本较高，模型升级后需要重新微调；二是 RAG 的方式更加灵活，可以通过更新知识库来动态调整 AI 的知识范围。未来如果有特定场景（如固定的输出格式要求），我们会考虑微调。

### RAG 实现类

**Q11: 你们为什么选择 ChromaDB？和其他向量数据库（如 Pinecone、Weaviate）相比如何？**
A: ChromaDB 是轻量级的嵌入式向量数据库，部署简单、零运维成本，适合我们的项目规模。Pinecone 是云服务，依赖网络且涉及数据外传；Weaviate 功能强大但对资源要求更高。ChromaDB 在功能和轻量性之间取得了很好的平衡，特别适合 FinPilot 的 MVP 阶段。

**Q12: 文档切片（Chunking）的策略是什么？**
A: 我们采用分层切片策略。先按文档的自然结构（标题、段落）进行粗粒度切片，确保每个 Chunk 的语义完整性。对于过长的段落，再按 Token 数（约 500 tokens）进行细粒度分割，Chunk 之间保留 10% 的重叠，避免信息断裂。

**Q13: 向量检索的准确率如何？如何优化？**
A: 当前 Top-5 召回率在 85% 以上。优化策略包括：（1）选择合适的 Embedding 模型；（2）调整 Chunk 大小和重叠率；（3）结合关键字检索（Hybrid Search）提升准确率；（4）对召回结果进行重排序（Re-ranking）。

**Q14: 如何处理多个用户上传相同文档的情况？**
A: 我们通过文档指纹（基于内容的 Hash）来判断文档是否已存在。如果已存在，直接复用已有的向量化结果，避免重复计算。同时每个用户的文档有独立的访问权限控制，确保数据隔离。

**Q15: 支持哪些文档格式？未来计划支持哪些？**
A: 目前支持 TXT、Markdown、PDF 格式。未来计划支持 Word（.docx）、Excel（.xlsx）、PowerPoint（.pptx）、图片中的文字提取（OCR）等格式，覆盖更广泛的文档类型。

### Multi-Agent 设计类

**Q16: 你们的 Agent 是如何通信和协作的？**
A: 采用 Orchestrator Pattern。Orchestrator Agent 接收用户请求，通过任务分解生成子任务列表，然后并行或串行调用各个子 Agent。每个子 Agent 独立完成任务后将结果返回给 Orchestrator，Orchestrator 将多源结果整合为最终输出。Agent 之间不直接通信，所有通信通过 Orchestrator 中转，保持架构的简洁性。

**Q17: Agent 的 Prompt 是如何设计的？**
A: 每个 Agent 有独立的 System Prompt，定义了其角色、能力边界、输出格式和工作流程。例如 Financial Analyst Agent 的 Prompt 规定了它必须输出财务指标分析、估值判断和基本面评价三个部分。Prompt 设计遵循"角色清晰、边界明确、输出结构化"的原则。

**Q18: 如何保证多个 Agent 输出的一致性？**
A: 通过三个层次保证：（1）Orchestrator 在任务分解时提供统一的上下文信息；（2）每个 Agent 的 Prompt 中规定了标准化的输出格式；（3）Orchestrator 在结果聚合阶段进行格式统一和风格对齐，确保最终输出的一致性。

**Q19: Agent 的执行是串行还是并行？**
A: 主要采用并行执行。对于相互独立的子任务（如基本面分析和技术面分析），Orchestrator 会同时调用多个 Agent，大幅缩短总响应时间。对于有依赖关系的任务（如先生成数据再撰写报告），则采用串行执行。

**Q20: 如果某个 Agent 执行失败，系统如何处理？**
A: 我们设计了超时机制和重试策略。如果某个 Agent 在超时时间内未返回结果，Orchestrator 会进行最多 2 次重试。如果仍然失败，Orchestrator 会跳过该 Agent 的结果，并在最终报告中标注"该部分分析暂时不可用"，同时记录日志供排查。

### 安全与扩展类

**Q21: 用户的数据如何保证安全？**
A: 从三个层面保障：（1）传输安全 — 所有 API 通信使用 HTTPS 加密；（2）存储安全 — 用户文档加密存储，向量数据无法反向还原原文；（3）权限安全 — 用户间数据完全隔离，每个用户只能访问自己的知识库和对话记录。

**Q22: 系统能否支持高并发？**
A: 当前架构通过水平扩展后端实例可以支持一定规模的并发。主要瓶颈在 AI API 的调用频率限制。解决方案包括：接入多个 API Key 做负载均衡、增加本地缓存层减少重复调用、未来支持模型本地部署。

**Q23: 如何保证系统的可扩展性？**
A: 通过模块化设计保证可扩展性。新增 Agent 只需实现 Agent 接口并注册到 Orchestrator；新增知识库类型只需实现新的文档解析器；新增前端页面只需在路由表中注册。Monorepo 架构让新增模块的集成成本降到最低。

**Q24: 系统有日志和监控机制吗？**
A: 有的。我们使用 Server-Sent Events 记录了所有 API 调用和 Agent 执行的日志，包括请求参数、响应时间、Token 消耗等关键指标。这些日志用于排查问题和持续优化系统性能。

**Q25: 你们如何处理敏感金融信息的合规问题？**
A: 首先，所有用户数据不在服务端持久化存储敏感信息（如交易密码等）。其次，RAG 知识库中的文档访问权限严格隔离。最后，AI 生成内容会标注"仅供学习参考，不构成投资建议"，明确免责声明。

### 商业与竞争类

**Q26: 这个项目的商业模式是什么？**
A: 计划采用 Freemium 模式。免费版提供基础的 AI 对话和有限的课程内容。付费版按月/年订阅，提供完整的股票分析、无限知识库容量、高级课程等增值功能。未来也考虑面向金融机构提供企业版，支持私有化部署和定制化 Agent。

**Q27: 你们的主要竞争对手有哪些？差异优势是什么？**
A: 主要竞争对手包括：通用 AI 助手（ChatGPT 等）、传统金融终端（Wind 等）、金融学习平台（Coursera 等）。我们的差异化优势在于"一体化"定位 — 将学习、研究、知识管理整合在一个平台，同时 Multi-Agent 架构针对金融场景进行了深度优化，这是目前任何竞品都没有做到的。

**Q28: 项目的目标市场规模是多少？**
A: 目标市场分为三层：（1）核心层 — 中国金融学习者（高校金融专业约 100 万在校生 + CFA/CPA 考生约 50 万）；（2）扩展层 — 金融从业者（中国约 800 万金融从业人员）；（3）想象层 — 个人投资者（中国约 2 亿股民）。初期聚焦核心层，验证产品价值后再向扩展层渗透。

### 团队与开发类

**Q29: 项目开发周期多长？团队规模如何？**
A: FinPilot 项目在 TRAE IDE 中开发完成，充分利用了 AI 辅助编程能力。前端和后端核心功能开发周期约 4-6 周，AI 层集成和优化约 2-3 周。项目由小型团队完成（2-3 名开发者），TRAE 的 AI 编程助手大幅提升了开发效率。

**Q30: 你们在 TRAE IDE 中开发的最大收获是什么？**
A: 最大的收获是 TRAE 的 AI 编程能力让我们能够以极小的团队完成了一个完整的前后端+AI 项目。具体来说：（1）代码生成效率高，尤其是前后端接口代码和类型定义；（2）上下文理解能力强，能够基于项目结构给出准确的代码建议；（3）调试辅助功能帮助快速定位问题。TRAE 让我们真正体验到了"AI 辅助编程"的生产力提升。
