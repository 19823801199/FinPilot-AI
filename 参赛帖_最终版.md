## 1. Demo 简介

### 产品是什么？

FinPilot AI 是一个面向金融学习与投资研究场景的 AI 智能工作台。

它把 **AI 学习助手、AI 股票研究、AI 知识库、Multi-Agent 专家协作** 集成到同一个平台，让用户能够完成从学习、分析到知识管理的完整流程。

### 面向哪些用户？

- **金融专业学生**：备考 431 金融学综合，需要系统化学习路径和智能练习
- **431 金融专硕考研党**：需要针对性的知识点讲解和错题追踪
- **个人投资者**：需要快速获取股票分析、研报解读和投资建议
- **希望利用 AI 提高学习和研究效率的人**

### 核心功能

**📚 AI 学习助手**

针对金融学习场景，提供课程学习、知识点讲解、练习题、错题整理等功能，帮助用户建立系统化学习路径。

![学习中心](docs/screenshots/learn.png)

**📈 AI 股票研究员**

集成 AKShare 真实 A 股行情数据，支持股票搜索、K 线图分析、AI 深度研究报告生成。AI 基于 DeepSeek 大模型，涵盖基本面分析、财务数据解读、行业对比、投资建议，通过 SSE 流式实时输出。

![股票研究员](docs/screenshots/stocks.png)

**📖 AI 知识库 RAG**

支持 PDF、Markdown 等文档上传，通过 ChromaDB 向量数据库进行语义存储。用户通过自然语言查询，AI 基于检索到的文档片段生成精准回答，并标注信息来源。完整的检索增强生成管线：文档上传 → 切块 → Embedding → 向量存储 → 语义检索 → 上下文组装 → LLM 生成 → 来源标注。

![知识库](docs/screenshots/knowledge.png)

**🤖 Multi-Agent 专家协作**

这是整个项目最核心的能力。系统内置 4 位 AI 专家，每位专家负责不同任务：

- **学习导师**：负责知识点讲解和学习规划
- **股票分析师**：负责行情分析和投资研究
- **知识专家**：负责文档问答和知识检索
- **报告整理专家**：负责结果整合和报告生成

用户只需要提出一个问题，系统就会自动规划任务、调用对应专家协同完成分析。

## 2. Demo 创作思路

### 灵感来源

这个项目来源于我自己的真实需求。

在学习金融和关注市场过程中，我发现：学习、研究、知识整理分别需要使用很多不同的软件。例如：

- 学习课程需要一个平台
- 查股票需要另一个平台
- 阅读研报又需要第三个平台
- AI 对话还需要切换到新的工具

工具越来越多，但效率却越来越低。

于是我开始思考：是否可以利用 AI，把这些原本分散的流程整合到一个平台中？

FinPilot AI 就是在这样的背景下诞生的。

### 希望解决的问题

- **痛点一：学习资源分散**。学习过程中需要不断切换教材、课程、笔记、练习题，缺少统一入口。431 金融学综合涵盖货币银行学、公司理财、投资学三门课程，知识点超过 200 个，考生需要在教材、网课、真题之间反复切换，缺乏一体化的学习平台。
- **痛点二：研究工具割裂**。投资者分析一只股票，需要打开行情软件看 K 线、打开研报平台看分析、打开 AI 工具做总结。三个工具之间的信息无法打通，分析效率低下。
- **痛点三：知识管理困难**。研究人员积累的 PDF 研报、笔记文档越来越多，传统文件夹管理方式无法支持语义检索。想找到某个观点的出处，往往需要翻阅数十份文档。

### 为什么选择这个方向？

金融学习和投资研究都属于专业场景。相比通用 AI，这类用户更加需要：更准确的知识、更完整的分析、更系统的学习体验。

金融是 AI 落地的高价值场景。一方面，金融知识具有专业性强、时效性高的特点，通用 AI 难以给出精准回答；另一方面，金融领域的用户付费意愿强，商业化路径清晰。更重要的是，这个方向将 AI 的三种核心能力（对话、检索、分析）与金融的真实需求深度结合，而非简单的"套壳"应用。

选择这个方向还有一个个人原因：我亲历过 431 备考的全过程，这本身就是一个"我为自己开发"的产品，每个功能都是我真的需要的东西。这让它不只是一个参赛作品，更是一个真实的需求落地。

## 4. TRAE 实践过程

### 开发完整流程

FinPilot AI 的开发分为 10 个阶段，全部在 TRAE 中完成。以下记录不是项目总结，而是真实的开发过程复盘。

---

#### 阶段1：需求分析

**1. 初始输入**

我当时只有一个模糊的想法："我想做一个 AI 金融助手，帮自己学习金融和分析股票。"完全不知道应该包含哪些功能、面向谁、做成什么样子。

**2. TRAE 如何拆解问题**

TRAE 根据我的输入，将这个模糊目标拆解为 3 个子任务：

- 子任务 A：识别目标用户画像（谁会用这个产品？）
- 子任务 B：梳理核心痛点（用户现在怎么解决这些问题的？有什么不方便？）
- 子任务 C：定义功能边界（MVP 应该包含哪些功能，哪些可以后续再加？）

TRAE 的拆解逻辑是：先做用户调研 → 再定义痛点 → 最后确定功能优先级。这样避免了我一开始就陷入"想做什么功能"的细节里。

**3. 实际实现过程**

通过多轮对话，TRAE 引导我回答了以下问题：我自己备考 431 时遇到什么困难？查股票时用了哪些工具？阅读研报后怎么管理？

最终产出：3 类目标用户（金融学生、考研党、个人投资者）、3 大痛点（学习资源分散、研究工具割裂、知识难以沉淀）、4 个核心功能模块（AI 学习、股票研究、知识库 RAG、Multi-Agent 协作），以及 MVP 范围（6 个页面、31 个 API 端点）。

**4. 关键问题 & 修复过程**

**卡点**：我一开始想把所有功能都塞进去，包括实时行情推送、社交讨论区、新闻聚合。TRAE 分析后指出这些功能会显著增加开发复杂度，建议第一轮只保留"AI 学习 + 股票研究 + 知识库"三个核心模块，Multi-Agent 作为增值功能放在第二轮。

**修复**：我接受了 TRAE 的建议，把 MVP 边界收敛到"能用、能演示、能闭环"的范围。这个决策后来证明是正确的——如果一开始就做太多，到现在可能一个完整功能都做不出来。

**5. 输出结果**

确定了产品方向、用户画像、功能优先级和 MVP 范围，为后续开发提供了清晰的路线图。

---

#### 阶段2：产品设计

**1. 初始输入**

需求已经明确，但我不知道怎么把这些功能组织成一个可交互的产品。我需要一份 PRD（产品需求文档），包含页面结构、交互流程和 API 设计。

**2. TRAE 如何拆解问题**

TRAE 将产品设计拆解为 4 个层次：

- 信息架构层：6 个页面怎么组织（工作台 / 学习中心 / 股票研究员 / 知识库 / 设置 / 404）
- 交互流程层：每个页面的核心操作流程（如"搜索股票 → 查看详情 → AI 分析"）
- 状态流转层：Multi-Agent 的 8 步状态机（意图识别→任务规划→任务拆分→分配专家→执行→收集结果→合并→最终回答）
- 接口定义层：前后端通信的 31 个 API 端点

**3. 实际实现过程**

TRAE 根据阶段1的需求，直接生成了 PRD 文档的初稿。我逐页审阅，提出修改意见（例如"工作台的左侧导航应该包含什么"、"股票详情页需要哪些数据卡片"），TRAE 根据反馈迭代。

**4. 关键问题 & 修复过程**

**卡点**：最初的 API 设计中，所有 AI 功能都走同一个 `POST /chat` 端点，通过消息内容来区分意图。但 TRAE 在生成后端代码时指出，这种方式会导致：

- 后端需要在前端做大量意图解析，职责不清
- 股票分析、知识库问答、普通聊天的参数结构完全不同，用一个端点会导致类型混乱
- 错误处理困难，前端无法区分是"AI 服务故障"还是"股票数据获取失败"

**修复**：TRAE 建议拆分为 4 个独立端点：`/chat/stream`（SSE 流式对话）、`/agent/orchestrator`（Multi-Agent 调度）、`/knowledge/query`（RAG 问答）、`/stocks/analyze`（股票分析）。这样前后端职责清晰，错误处理也更有针对性。

**5. 输出结果**

产出完整 PRD 文档：6 个页面交互流程、Multi-Agent 8 步状态机、31 个 API 端点接口定义。

#### 阶段3：UI 设计

**1. 初始输入**

PRD 已定，但我对 UI 没有明确想法。我知道想要"深色风格、金融感、专业但不冰冷"，但不知道怎么落地。

**2. TRAE 如何拆解问题**

TRAE 将 UI 设计拆解为 3 个层面：

- 设计系统层：颜色、字体、间距、圆角等基础 token
- 组件层：Button、Card、Input、Badge 等 14 个基础组件的规范
- 布局层：6 个页面的整体布局结构（侧边栏 + 主内容区 + 状态面板）

**3. 实际实现过程**

TRAE 最初建议使用 shadcn/ui 作为组件库基础，因为这样能快速搭建。但在生成配色方案时，我发现 shadcn 的默认主题很难调出"金融科技感"，而且组件体积较大。

**4. 关键问题 & 修复过程**

**卡点**：shadcn/ui 的默认组件风格偏"后台管理系统"，不够有金融产品的专业感。而且它的主题配置是通过 CSS 变量覆盖，TailwindCSS v4 的 `@theme` 指令和 shadcn 的变量系统有冲突。

**修复**：TRAE 分析后建议放弃 shadcn/ui，直接用 TailwindCSS v4 的 `@theme` 指令从零构建设计系统。这样虽然前期工作量更大，但配色完全可控。

TRAE 生成了四级背景层次（#08090d → #111827 → #161922 → #1e293b）和三色强调系统（翡翠绿 #00e5a0、紫色 #6366f1、琥珀色 #f59e0b），并定义了 14 个基础组件规范。这个调整让后续的页面开发非常顺畅——所有组件风格统一，没有 shadcn 的"默认感"。

**5. 输出结果**

生成了完整的设计系统：TailwindCSS v4 `@theme` 配置、14 个基础 UI 组件规范、6 个页面布局模板。

#### 阶段4：数据库设计

**1. 初始输入**

需要设计前后端统一的数据模型。我原本想直接用 PostgreSQL，因为"看起来更像生产环境"。

**2. TRAE 如何拆解问题**

TRAE 将数据库设计拆解为：

- 核心实体识别：User、Conversation、Message、Course、Exercise、Stock、Document、Agent、TaskPlan
- 关系梳理：一对多、多对多关系
- 前后端类型映射：Pydantic Schema（后端）↔ TypeScript Interface（前端）

**3. 实际实现过程**

TRAE 根据 PRD 自动推导了 30 个数据模型。但在这个过程中，我发现了一个问题。

**4. 关键问题 & 修复过程**

**卡点**：我一开始坚持用 PostgreSQL，TRAE 也生成了 PostgreSQL 的连接配置和 Alembic 迁移脚本。但在实际运行时，我发现：

- 本地开发每次改模型都要跑迁移，很费时间
- PostgreSQL 需要额外安装和配置，增加了启动门槛
- 比赛评审时评委不一定有 PostgreSQL 环境

**修复**：TRAE 建议改为 SQLite（开发模式）+ PostgreSQL（生产模式）的双模式设计。开发时用 SQLite（零配置、单文件、随删随建），生产部署时再切换到 PostgreSQL。这个方案后来也体现在 `config.py` 中——`DATABASE_URL` 默认指向 SQLite，注释中预留了 PostgreSQL 配置。

**5. 输出结果**

产出 30 个数据模型：后端 6 个 Pydantic Schema 文件、前端 5 个 TypeScript 类型文件。核心设计包括泛型 `ApiResponse<T>` 统一响应格式、Agent 角色枚举、DAG 任务依赖字段。

#### 阶段5：工程初始化

**1. 初始输入**

需要搭建项目骨架：Monorepo 结构、前后端分离、Docker 支持、环境变量管理。

**2. TRAE 如何拆解问题**

TRAE 将工程初始化拆解为：

- 目录结构：`apps/web`（Next.js）、`apps/api`（FastAPI）、`docker/`（构建配置）、`docs/`（文档和截图）
- 前端配置：Next.js 15 + TailwindCSS v4 + TypeScript 严格模式
- 后端配置：FastAPI + Uvicorn + Pydantic v2
- 部署配置：Docker 多阶段构建、docker-compose

**3. 实际实现过程**

TRAE 生成了完整的 Monorepo 骨架和配置文件。但在安装前端依赖时遇到了问题。

**4. 关键问题 & 修复过程**

**卡点**：`npm install` 失败，报错 `ERESOLVE could not resolve`。原因是 Next.js 15 依赖 React 19（peer dependency），但项目中显式安装了 React 18，npm 的严格依赖解析拒绝安装。

**TRAE 分析**：npm v7+ 的 peer dependency 自动安装机制和 Next.js 15 的声明有冲突。TRAE 提出了 3 个解决方案：

1. 升级 React 到 19（风险：可能影响其他依赖的兼容性）
2. 使用 `--legacy-peer-deps` 跳过严格检查（风险：可能引入运行时错误）
3. 手动锁定版本，用 `overrides` 强制指定 React 18

**修复**：我选择了方案 3，在 `package.json` 中添加 `overrides` 字段强制使用 React 18。TRAE 同时验证了所有依赖在 React 18 下的兼容性，确认没有运行时问题。这个修复后来也写进了启动器的 fallback 逻辑中——如果 `npm install` 失败，自动重试 `--legacy-peer-deps`。

**5. 输出结果**

Monorepo 项目骨架搭建完成，包含完整的前后端配置、Docker 构建文件、环境变量模板（.env.example，11 个变量）。

#### 阶段6：前端开发

**1. 初始输入**

需要根据设计系统实现 6 个页面：工作台、学习中心、股票研究员、知识库、设置、404。

**2. TRAE 如何拆解问题**

TRAE 将前端开发拆解为：

- 状态管理：18 个 Zustand Store（按功能域拆分，避免单个 store 过大）
- 组件层：30+ 个 React 组件，按 atomic design 分层
- 数据流：API 调用 → Store 更新 → UI 响应
- 特殊功能：SSE 流式输出接收（这是前端最复杂的部分）

**3. 实际实现过程**

TRAE 逐页生成代码。我主要负责审阅和调整交互细节（例如"发送按钮的 loading 状态怎么显示"、"错误提示用什么颜色"）。

SSE 流式输出是前端开发中技术难度最高的部分。TRAE 帮我设计了完整的接收逻辑：使用 `fetch` + `ReadableStream` + `TextDecoder` 逐行解析 SSE 格式数据，实时更新 UI。

**4. 关键问题 & 修复过程**

**卡点**：SSE 连接在组件卸载时没有被正确关闭，导致内存泄漏和"幽灵请求"。具体表现是：用户离开工作台页面后再返回，会看到上一次的流式输出还在继续，甚至同时有多个连接在跑。

**TRAE 分析**：问题在于 `useEffect` 的 cleanup 函数没有正确取消 `fetch` 请求。`fetch` 返回的 `Response.body` 是一个 `ReadableStream`，必须通过 `reader.cancel()` 来中断，但之前的代码只关闭了 reader，没有中止底层的 HTTP 请求。

**修复**：TRAE 引入了 `AbortController` 方案。每次发起 SSE 请求时创建一个新的 `AbortController`，将 `signal` 传给 `fetch`。在组件卸载或用户点击"停止生成"时，调用 `controller.abort()` 彻底中断请求。同时 cleanup 函数中依次执行 `reader.cancel()` → `reader.releaseLock()` → `controller.abort()`，确保连接完全释放。

这个修复大幅提升了前端的稳定性，之后再也没出现过 SSE 泄漏问题。

**5. 输出结果**

6 个页面全部实现，30+ 个 React 组件、18 个 Zustand Store、完整的 SSE 流式输出前端接收逻辑。

#### 阶段7：后端开发

**1. 初始输入**

需要根据 PRD 实现 31 个 API 端点，覆盖健康检查、AI 对话、股票分析、知识库管理、学习系统、Multi-Agent 协作。

**2. TRAE 如何拆解问题**

TRAE 将后端开发拆解为：

- 路由层：6 个 FastAPI Router 模块（health、chat、stock、learning、knowledge、agent）
- 服务层：20 个业务服务类，按领域拆分
- 基础设施层：AI 客户端封装（DeepSeek API）、数据库连接、配置管理
- 异常处理：统一错误响应格式、指数退避重试

**3. 实际实现过程**

TRAE 按模块逐一实现。我主要参与 API 接口的评审，确保前后端数据格式一致。

**4. 关键问题 & 修复过程**

**卡点**：前后端数据模型命名不一致，导致前端 TypeScript 类型报错。例如后端返回的字段叫 `created_at`（snake_case），但前端接口定义写的是 `createdAt`（camelCase）。这种不一致在开发初期并不明显，但在联调时大量暴露。

**TRAE 分析**：根本原因是阶段4的数据库设计时，前后端类型文件是分开生成的，没有统一的命名规范。TRAE 建议引入一个"类型契约"——后端 Pydantic Schema 使用 `ConfigDict` 的 `alias_generator` 自动转换，前端统一使用 camelCase。

**修复**：TRAE 批量修改了后端的 Schema 文件，添加 `alias_generator=lambda x: to_camel(x)`，同时更新前端的 TypeScript 类型定义。这个调整虽然花了一些时间，但彻底消除了后续联调中的类型不一致问题。

**5. 输出结果**

6 个 API 路由模块、20 个业务服务类、完整的 AI 客户端封装（含指数退避重试）。31 个 API 端点全部实现。

#### 阶段8：RAG 开发

**1. 初始输入**

需要实现完整的检索增强生成管线：文档上传 → 切块 → Embedding → 向量存储 → 语义检索 → 上下文组装 → LLM 生成 → 来源标注。

**2. TRAE 如何拆解问题**

TRAE 将 RAG 拆解为 5 个服务：

- Chunk Service：文档切块（500 字符/块，50 字符重叠）
- Embedding Service：文本向量化
- Chroma Service：向量存储（支持 PersistentClient 和内存 Fallback）
- Retriever Service：语义检索（Top-K 召回）
- RAG Service：主服务（检索 → 组装上下文 → LLM 生成 → 来源标注）

**3. 实际实现过程**

TRAE 实现了完整的管线代码。但第一次联调时，RAG 的回答质量很差。

**4. 关键问题 & 修复过程**

**卡点**：RAG 问答的回答质量很低，经常出现"根据文档内容..."这种空洞回答，或者完全忽略检索到的文档片段、直接用 LLM 的预训练知识回答。

**TRAE 分析**：问题出在 Prompt 设计上。最初的 Prompt 只是简单地把检索到的文档片段拼接后发给 LLM，没有明确告诉 LLM"必须基于提供的文档回答"和"如果文档中没有相关信息，请明确说明"。

**修复过程（3 轮迭代）**：

- **第 1 轮**：TRAE 在 system_prompt 中加入指令"你必须基于以下文档片段回答用户问题。如果文档中没有相关信息，请明确说明'根据现有资料无法回答'。"效果：LLM 开始引用文档，但回答过于机械，经常大段复制原文。
- **第 2 轮**：TRAE 调整 Prompt，要求 LLM"用自己的话总结文档内容，不要直接复制原文。回答要简洁，控制在 200 字以内。"效果：回答更自然了，但偶尔会出现"幻觉"——加入文档中没有的信息。
- **第 3 轮**：TRAE 在 Prompt 中加入"来源标注"要求：每句话后面用 `[来源: 文档名]` 标注信息出处。同时在后端的 RAG Service 中加入了"答案-文档相似度校验"，如果 LLM 的回答和检索到的文档片段相似度低于阈值，就返回"无法根据现有资料回答"。

经过 3 轮 Prompt 迭代，RAG 的回答质量从"几乎不可用"提升到了"可以直接用于参考"的水平。

**5. 输出结果**

完整的检索增强生成管线，包含文档切块、向量存储、语义检索、LLM 生成和来源标注。RAG 回答质量经过 3 轮 Prompt 迭代达到可用标准。

#### 阶段9：Multi-Agent 开发

**1. 初始输入**

需要实现项目最核心的能力：4 位 AI 专家协同工作，Orchestrator 自动调度。

**2. TRAE 如何拆解问题**

TRAE 将 Multi-Agent 拆解为 3 个核心组件：

- Agent Manager：4 位专家的状态管理和角色配置
- Orchestrator Service：意图识别 → 任务规划 → DAG 拓扑排序 → 分配专家 → 结果合并
- Workflow Manager：工作流执行追踪和状态同步

**3. 实际实现过程**

这是整个项目中技术难度最高的部分。TRAE 先帮我设计了 4 位专家的 system_prompt 和 task_prompt，然后实现了 Orchestrator 的调度逻辑。

**4. 关键问题 & 修复过程**

**卡点**：Multi-Agent 的第一次测试结果是灾难性的。我输入"分析贵州茅台"，Orchestrator 识别为 stock_analysis，分配了研究分析师和报告撰写专家。但输出结果是：

- 研究分析师返回了一段零散的数据罗列（没有分析结论）
- 报告撰写专家直接复制了研究分析师的内容，没有进行整合
- 两位专家的回答风格不一致，合并后的报告读起来像两个人在争吵

**TRAE 分析**：根本问题是 Prompt 没有定义清楚"每位专家应该输出什么格式"。研究分析师的 Prompt 只说了"分析股票"，但没有要求"输出结构化报告，包含基本面、技术面、投资建议三个部分"。报告撰写专家的 Prompt 也没有说明"如何整合多位专家的结果"。

**修复过程（4 轮迭代）**：

- **第 1 轮**：TRAE 为每位专家设计了明确的输出格式模板。例如研究分析师必须按"基本面分析 / 财务数据 / 行业对比 / 投资建议"四个章节输出，每章不超过 150 字。
- **第 2 轮**：在 Orchestrator 的合并 Prompt 中加入"整合要求"："请综合以上各位专家的分析，生成一份结构清晰、逻辑连贯的投资分析报告。不要简单拼接，要提炼核心观点。"
- **第 3 轮**：加入"风格统一"要求。因为 DeepSeek 的每次调用独立，不同专家的回答风格可能差异很大。TRAE 在 system_prompt 中加入统一的语气要求："使用专业但易懂的语言，避免过于学术化的表达。"
- **第 4 轮**：加入"自校验"机制。Orchestrator 在合并结果后，会额外调用一次 LLM 进行"质量检查"：检查报告是否完整、是否有矛盾、是否包含投资建议。如果检查不通过，会自动要求相关专家重新生成。

经过 4 轮迭代，Multi-Agent 的输出从"零散的数据罗列"变成了"结构化的投资分析报告"，可以直接用于参考。

**5. 输出结果**

4 位金融专家（学习导师、研究分析师、知识专家、报告撰写员）各配备独立的 system_prompt 和 task_prompt。Orchestrator 调度中心实现完整工作流，Prompt 工程经过 4 轮迭代达到可用标准。
