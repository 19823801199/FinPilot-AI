## 1. Demo 简介

### 产品是什么？

FinPilot AI 是一个面向金融学习与投资研究场景的 AI 智能工作台。

它把 **AI 学习助手、AI 股票研究、AI 知识库、Multi-Agent 专家协作** 集成到同一个平台，让用户能够完成从学习、分析到知识管理的完整流程。

### 面向哪些用户？

- **金融专业学生**：备考 431 金融学综合，需要系统化学习路径和智能练习
- **431 金融专硕考研党**：需要针对性的知识点讲解和错题追踪
- **个人投资者**：需要快速获取股票分析、研报解读和投资建议
- **希望利用 AI 提高学习和研究效率的人**

### 核心功能

**📚 AI 学习助手**

针对金融学习场景，提供课程学习、知识点讲解、练习题、错题整理等功能，帮助用户建立系统化学习路径。

![学习中心](docs/screenshots/learn.png)

**📈 AI 股票研究员**

集成 AKShare 真实 A 股行情数据，支持股票搜索、K 线图分析、AI 深度研究报告生成。AI 基于 DeepSeek 大模型，涵盖基本面分析、财务数据解读、行业对比、投资建议，通过 SSE 流式实时输出。

![股票研究员](docs/screenshots/stocks.png)

**📖 AI 知识库 RAG**

支持 PDF、Markdown 等文档上传，通过 ChromaDB 向量数据库进行语义存储。用户通过自然语言查询，AI 基于检索到的文档片段生成精准回答，并标注信息来源。完整的检索增强生成管线：文档上传 → 切块 → Embedding → 向量存储 → 语义检索 → 上下文组装 → LLM 生成 → 来源标注。

![知识库](docs/screenshots/knowledge.png)

**🤖 Multi-Agent 专家协作**

这是整个项目最核心的能力。系统内置 4 位 AI 专家，每位专家负责不同任务：

- **学习导师**：负责知识点讲解和学习规划
- **股票分析师**：负责行情分析和投资研究
- **知识专家**：负责文档问答和知识检索
- **报告整理专家**：负责结果整合和报告生成

用户只需要提出一个问题，系统就会自动规划任务、调用对应专家协同完成分析。

## 2. Demo 创作思路

### 灵感来源

这个项目来源于我自己的真实需求。

在学习金融和关注市场过程中，我发现：学习、研究、知识整理分别需要使用很多不同的软件。例如：

- 学习课程需要一个平台
- 查股票需要另一个平台
- 阅读研报又需要第三个平台
- AI 对话还需要切换到新的工具

工具越来越多，但效率却越来越低。

于是我开始思考：是否可以利用 AI，把这些原本分散的流程整合到一个平台中？

FinPilot AI 就是在这样的背景下诞生的。

### 希望解决的问题

- **痛点一：学习资源分散**。学习过程中需要不断切换教材、课程、笔记、练习题，缺少统一入口。431 金融学综合涵盖货币银行学、公司理财、投资学三门课程，知识点超过 200 个，考生需要在教材、网课、真题之间反复切换，缺乏一体化的学习平台。
- **痛点二：研究工具割裂**。投资者分析一只股票，需要打开行情软件看 K 线、打开研报平台看分析、打开 AI 工具做总结。三个工具之间的信息无法打通，分析效率低下。
- **痛点三：知识管理困难**。研究人员积累的 PDF 研报、笔记文档越来越多，传统文件夹管理方式无法支持语义检索。想找到某个观点的出处，往往需要翻阅数十份文档。

### 为什么选择这个方向？

金融学习和投资研究都属于专业场景。相比通用 AI，这类用户更加需要：更准确的知识、更完整的分析、更系统的学习体验。

金融是 AI 落地的高价值场景。一方面，金融知识具有专业性强、时效性高的特点，通用 AI 难以给出精准回答；另一方面，金融领域的用户付费意愿强，商业化路径清晰。更重要的是，这个方向将 AI 的三种核心能力（对话、检索、分析）与金融的真实需求深度结合，而非简单的"套壳"应用。

选择这个方向还有一个个人原因：我亲历过 431 备考的全过程，这本身就是一个"我为自己开发"的产品，每个功能都是我真的需要的东西。这让它不只是一个参赛作品，更是一个真实的需求落地。

### 本地部署（完整体验推荐）

本地部署可以体验全部 AI 功能（对话、流式输出、Multi-Agent 协作、股票分析、知识库 RAG）。

**前置条件**：Node.js >= 20、Python >= 3.12、DeepSeek API Key

**配置 API Key**：

```bash
# 1. 克隆仓库
git clone https://github.com/19823801199/FinPilot-AI.git
cd FinPilot-AI

# 2. 复制环境变量模板并填入 DeepSeek API Key
cp .env.example .env
# 编辑 .env 文件，将 DEEPSEEK_API_KEY 替换为你的真实 Key
```

> AI 功能（对话、股票分析、知识库问答）依赖 DeepSeek API。你可以在 [platform.deepseek.com](https://platform.deepseek.com) 免费注册获取 API Key。

**启动后端**：

```bash
cd apps/api
python -m venv venv
venv\Scripts\activate        # Windows
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

**启动前端**：

```bash
cd apps/web
npm install
npm run dev
```

打开 http://localhost:3000 即可体验全部功能。

## 4. TRAE 实践过程

### 开发完整流程

FinPilot AI 的开发分为 10 个阶段，全部在 TRAE 中完成。以下记录不是项目总结，而是我和 TRAE 对话的真实过程复盘。

---

#### 阶段1：需求分析

**1. 初始输入（我说了什么）**

我的第一条输入是：

> "我想做一个 AI 金融助手，帮自己学习金融和分析股票。"

就这么一句话，完全不知道应该包含什么功能、面向谁、做成什么形式。

**2. TRAE 拆解方式（它怎么理解）**

TRAE 没有直接给我功能列表，而是先反问了我几个问题：

> "你能描述一下你平时学习金融或分析股票时的具体流程吗？比如：你现在用什么工具？觉得哪里不方便？"

我回答了：备考 431 时需要不断切换教材和网课；查股票时要开行情软件、研报平台、AI 工具三个窗口；收藏的 PDF 很多但找不到了。

TRAE 接着追问：

> "你的目标用户只有你自己，还是也包括其他金融学习者或投资者？"

我回答应该包括和我一样的人——金融学生、考研党、个人投资者。

**3. Prompt 迭代过程**

- **第 1 轮**：我输入了模糊想法 → TRAE 追问具体场景 → 我描述了 3 个痛点
- **第 2 轮**：我要求"帮我整理出这个产品应该有哪些功能" → TRAE 生成了初版功能清单（AI 学习、股票研究、知识库、AI 对话）
- **第 3 轮**：我发现清单里有"实时行情推送""社交讨论区""新闻聚合"等过于复杂的功能，问 TRAE"这些是不是 MVP 必须的？" → TRAE 分析后指出这些会显著增加开发复杂度，建议第一轮只保留"AI 学习 + 股票研究 + 知识库"三个核心模块，Multi-Agent 作为增值功能放在第二轮

**4. 卡点与修复（最重要加分项）**

**卡点**：我想把所有功能都塞进第一轮，包括实时行情推送、社交讨论区、新闻聚合。

**修复过程**：TRAE 没有直接否定我，而是帮我算了一笔账——"实时行情推送需要 WebSocket 连接和行情数据源接入，社交讨论区需要用户系统和消息推送，新闻聚合需要爬虫和 NLP 处理。这些功能每个都需要独立的开发周期，如果全部放在第一轮，MVP 可能无法在比赛前完成。"

我接受了 TRAE 的建议，把 MVP 边界收敛到"能用、能演示、能闭环"的范围。这个决策后来证明是正确的。

**5. 输出结果**

3 类目标用户、3 大痛点、4 个核心功能模块、MVP 范围（6 个页面、31 个 API 端点）。

#### 阶段2：产品设计

**1. 初始输入（我说了什么）**

> "功能确定了，但我不知道怎么把这些功能组织成一个可交互的产品。我需要一份 PRD，包含页面结构、交互流程和 API 设计。"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 将我的需求拆解为 4 个层次：信息架构、交互流程、状态流转、接口定义。它没有直接生成 PRD，而是先问我：

> "你希望用户打开产品的第一眼看到什么？是一个总控面板，还是直接进入某个功能模块？"

我回答希望有一个"工作台"作为总入口，左侧导航切换不同功能。

**3. Prompt 迭代过程**

- **第 1 轮**：我描述"工作台 + 左侧导航"的想法 → TRAE 生成 PRD 初稿，包含 6 个页面（工作台 / 学习中心 / 股票研究员 / 知识库 / 设置 / 404）
- **第 2 轮**：我审阅 PRD，提出"工作台的左侧导航应该包含什么""股票详情页需要哪些数据卡片" → TRAE 根据反馈迭代，增加了导航项和数据卡片布局
- **第 3 轮**：我提出 Multi-Agent 的状态流转需要更详细的设计 → TRAE 补充了 8 步状态机（意图识别→任务规划→任务拆分→分配专家→执行→收集结果→合并→最终回答）

**4. 卡点与修复（最重要加分项）**

**卡点**：最初的 API 设计中，所有 AI 功能都走同一个 `POST /chat` 端点。我在和 TRAE 讨论后端实现时，TRAE 主动指出：

> "如果所有 AI 功能都走 /chat，后端需要在前端做大量意图解析，股票分析、知识库问答、普通聊天的参数结构完全不同，用一个端点会导致类型混乱，错误处理也困难。"

**修复过程**：TRAE 建议拆分为 4 个独立端点。我一开始犹豫"这样前端是不是要写更多代码？"，TRAE 解释说"虽然前端需要多几个 API 调用，但每个端点的职责清晰，错误处理更有针对性，而且 TypeScript 类型安全会得到极大提升。"

我接受了建议，最终确定 `/chat/stream`（SSE 流式对话）、`/agent/orchestrator`（Multi-Agent 调度）、`/knowledge/query`（RAG 问答）、`/stocks/analyze`（股票分析）四个独立端点。

**5. 输出结果**

完整 PRD 文档：6 个页面交互流程、Multi-Agent 8 步状态机、31 个 API 端点接口定义。

#### 阶段3：UI 设计

**1. 初始输入（我说了什么）**

> "我想要深色风格、有金融产品的专业感，但不要冰冷。怎么配色？"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 先给了我几个参考方向：

> "金融产品的 UI 通常有几种风格：A. 偏 Bloomberg 的专业数据终端风（深色+大量数据）；B. 偏 Robinhood 的简洁现代风（浅色+卡片）；C. 偏 TradingView 的图表主导风。你倾向哪种？"

我回答想要 A 和 B 的结合——深色专业感 + 简洁现代卡片布局。

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 最初建议使用 shadcn/ui，生成了一套基于 shadcn 的配色方案 → 我看了之后觉得"默认感太重，不够有金融科技的味道"
- **第 2 轮**：我反馈"shadcn 的蓝色主题太像后台管理系统了，我想要更有层次感的深色" → TRAE 建议放弃 shadcn，直接用 TailwindCSS v4 的 `@theme` 从零构建
- **第 3 轮**：TRAE 生成了四级背景层次（#08090d → #111827 → #161922 → #1e293b）和三色强调系统（翡翠绿 #00e5a0、紫色 #6366f1、琥珀色 #f59e0b） → 我确认"这个感觉对了"

**4. 卡点与修复（最重要加分项）**

**卡点**：shadcn/ui 的主题配置和 TailwindCSS v4 的 `@theme` 指令有冲突。我在实际使用时发现，shadcn 的 CSS 变量覆盖了 `@theme` 定义的颜色，导致页面上的 Button 和 Input 颜色不一致。

**修复过程**：我截图给 TRAE 看了问题，TRAE 分析后确认是"shadcn 的变量系统和 Tailwind v4 不兼容"。TRAE 给出了两个方案：A. 降级到 Tailwind v3 + shadcn；B. 完全放弃 shadcn，自己写组件。我选择了 B，因为"自己的设计系统更灵活"。TRAE 随后生成了 14 个基础组件的完整规范。

**5. 输出结果**

TailwindCSS v4 `@theme` 配置、14 个基础 UI 组件规范、6 个页面布局模板。

#### 阶段4：数据库设计

**1. 初始输入（我说了什么）**

> "帮我设计数据模型，前后端类型要对齐。"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 问了我几个关键问题：

> "你希望用什么数据库？SQLite（零配置）、PostgreSQL（生产级）、还是 MySQL？"

我回答"PostgreSQL 吧，看起来更专业"。

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 生成了基于 PostgreSQL 的 Pydantic Schema 和 Alembic 迁移脚本 → 我运行时发现每次改模型都要执行迁移，很麻烦
- **第 2 轮**：我问 TRAE"能不能开发时用 SQLite，生产再切 PostgreSQL？" → TRAE 同意，修改了 `config.py`，让 `DATABASE_URL` 默认指向 SQLite，注释中预留 PostgreSQL 配置
- **第 3 轮**：我要求"前后端字段命名要统一" → TRAE 在 Pydantic Schema 中加入了 `alias_generator`，自动将 snake_case 转为 camelCase

**4. 卡点与修复（最重要加分项）**

**卡点**：前后端类型文件分开生成，命名不一致。例如后端用 `created_at`，前端接口定义写成 `createdAt`，联调时大量 TypeScript 报错。

**修复过程**：我截了一张 TypeScript 错误截图给 TRAE，TRAE 分析后说"这是命名规范不统一导致的。建议后端用 alias_generator 自动转换，前端统一使用 camelCase。"TRAE 随后批量修改了所有 Schema 文件，彻底解决了这个问题。

**5. 输出结果**

30 个数据模型、SQLite/PostgreSQL 双模式配置、前后端类型自动转换。

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 生成 Monorepo 骨架 → 我运行 `npm install` 时失败，报错 `ERESOLVE could not resolve`
- **第 2 轮**：我把错误日志贴给 TRAE → TRAE 分析是 Next.js 15 的 peer dependency（React 19）和项目中的 React 18 冲突
- **第 3 轮**：TRAE 给出 3 个解决方案，我选择用 `overrides` 强制锁定 React 18 → TRAE 修改 `package.json` 并验证兼容性

**4. 卡点与修复（最重要加分项）**

**卡点**：`npm install` 失败，原因是 Next.js 15 声明了 React 19 的 peer dependency，但项目中显式安装了 React 18。

**修复过程**：TRAE 提出了 3 个方案，我选了最保守的——在 `package.json` 中加 `overrides` 字段强制使用 React 18。TRAE 验证后确认所有依赖在 React 18 下运行正常。这个修复后来也写进了启动器的 fallback 逻辑中。

**5. 输出结果**

Monorepo 骨架、前后端配置、Docker 构建文件、环境变量模板。

#### 阶段6：前端开发

**1. 初始输入（我说了什么）**

> "实现工作台页面，要有 SSE 流式输出，AI 回复要逐字显示。"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 将需求拆解为：

> "你需要两个核心组件：ChatInput（输入框 + 发送按钮）和 ChatArea（消息列表 + SSE 接收）。SSE 的实现是关键，需要处理连接生命周期。"

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 生成了 `chat-input.tsx` 和 SSE 接收逻辑 → 我测试发现流式输出正常，但切换页面后再回来，上一次的连接还在跑
- **第 2 轮**：我描述问题"组件卸载后 SSE 连接没关闭，导致内存泄漏" → TRAE 分析后认为是 `useEffect` cleanup 函数没有正确取消 `fetch` 请求
- **第 3 轮**：TRAE 引入 `AbortController` 方案，重写 SSE 接收逻辑 → 我测试确认问题已修复

**4. 卡点与修复（最重要加分项）**

**卡点**：SSE 连接泄漏。用户离开工作台页面后再返回，会看到上一次的流式输出还在继续，甚至同时有多个连接在跑。

**修复过程**：

我把控制台截图发给 TRAE，显示同时有 3 个 `GET /chat/stream` 请求在进行。TRAE 分析后指出：`fetch` 返回的 `ReadableStream` 必须通过 `reader.cancel()` 中断，但之前的 cleanup 函数只释放了 reader，没有中止底层的 HTTP 请求。

TRAE 重写了 SSE 接收逻辑：每次发起请求时创建 `AbortController`，将 `signal` 传给 `fetch`；cleanup 时依次执行 `reader.cancel()` → `reader.releaseLock()` → `controller.abort()`。我测试后确认连接完全释放。

**5. 输出结果**

6 个页面、30+ 个 React 组件、18 个 Zustand Store、SSE 流式输出（含 AbortController 防泄漏）。

**修复过程**：我截了一张满屏红色的 TypeScript 错误图给 TRAE。TRAE 说"这是命名规范不统一导致的，建议在 Pydantic Schema 中用 alias_generator 自动转换。"TRAE 随后批量修改了 6 个 Schema 文件，每个字段都加了 alias。我重新联调后，类型错误全部消除。

#### 阶段8：RAG 开发

**1. 初始输入（我说了什么）**

> "实现文档上传和 AI 问答，AI 要基于上传的文档回答，不要胡说。"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 拆解为 5 个服务：文档切块、Embedding、向量存储、语义检索、RAG 主服务。

> "RAG 的核心是'检索 → 组装上下文 → LLM 生成 → 来源标注'。我会先实现基础管线，然后我们一起调 Prompt。"

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 实现完整管线 → 我测试上传了一份 PDF，问"这篇文章的核心观点是什么？" → AI 回答"根据文档内容..."（空洞，没有实质内容）
- **第 2 轮**：我反馈"AI 在胡说，没有基于文档回答" → TRAE 分析是 Prompt 没有明确要求"必须基于文档" → 修改 system_prompt，加入强制引用指令
- **第 3 轮**：测试发现 AI 开始大段复制原文，像复读机 → TRAE 调整 Prompt，要求"用自己的话总结，不要直接复制" → 回答变自然了
- **第 4 轮**：偶尔出现"幻觉"（加入文档中没有的信息）→ TRAE 加入来源标注要求 + 答案-文档相似度校验

**4. 卡点与修复（最重要加分项）**

**卡点**：RAG 回答质量差，经历了"空洞 → 机械复制 → 幻觉"三个阶段的问题。

**修复过程**：

我把每轮测试的对话截图都发给了 TRAE。

- 第 1 轮测试后，TRAE 说"Prompt 没有给 LLM 明确的约束，它不知道必须基于文档回答。"于是加入了"你必须基于以下文档片段回答"的指令。
- 第 2 轮测试后，TRAE 说"LLM 在'安全地回答'——它怕说错，所以选择复制原文。需要明确要求它用自己的话总结。"
- 第 3 轮测试后，TRAE 说"LLM 在'过度发挥'——它试图让回答更完整，但加入了文档外的知识。需要加入来源标注，让每句话都可追溯。"

经过 3 轮 Prompt 迭代，RAG 从"几乎不可用"提升到"可以直接参考"。

**5. 输出结果**

完整 RAG 管线（切块 → Embedding → 向量存储 → 检索 → LLM 生成 → 来源标注），Prompt 经过 3 轮迭代。

#### 阶段9：Multi-Agent 开发

**1. 初始输入（我说了什么）**

> "实现多专家协作。我输入'分析贵州茅台'，系统应该自动分配专家、协同完成分析。"

**2. TRAE 拆解方式（它怎么理解）**

TRAE 拆解为 3 个核心组件：Agent Manager（4 位专家）、Orchestrator（调度中心）、Workflow Manager（执行追踪）。

> "这是项目最复杂的部分。我会先设计 4 位专家的 Prompt，然后实现 Orchestrator 的调度逻辑。"

**3. Prompt 迭代过程**

- **第 1 轮**：TRAE 实现 Orchestrator + 4 位专家 → 我输入"分析贵州茅台" → 输出灾难性：研究分析师罗列零散数据，报告撰写专家直接复制，风格不一致
- **第 2 轮**：我截图给 TRAE，描述"像两个人在争吵" → TRAE 分析是"Prompt 没有定义输出格式" → 为每位专家设计结构化模板
- **第 3 轮**：测试发现合并后的报告还是拼接感重 → TRAE 在合并 Prompt 中加入"提炼核心观点，不要简单拼接"
- **第 4 轮**：风格仍不统一 → TRAE 加入统一语气要求 + 自校验机制

**4. 卡点与修复（最重要加分项）**

**卡点**：Multi-Agent 第一次测试结果灾难性。研究分析师输出零散数据，报告撰写专家直接复制，合并后像两个人在争吵。

**修复过程**：

我把完整的输出对话发给 TRAE。TRAE 逐条分析：

> "研究分析师的 Prompt 只说了'分析股票'，但没有要求'输出结构化报告'。报告撰写专家的 Prompt 也没有说明'如何整合多位专家的结果'。这导致每个专家都在'自由发挥'，输出格式完全不统一。"

TRAE 的修复分 4 轮：

- **第 1 轮**：给每位专家设计输出模板（研究分析师必须按"基本面 / 财务数据 / 行业对比 / 投资建议"四章输出）
- **第 2 轮**：合并 Prompt 中加入"提炼核心观点，不要简单拼接"
- **第 3 轮**：统一语气要求"专业但易懂，避免学术化"
- **第 4 轮**：加入自校验机制，合并后额外调用 LLM 检查报告完整性

经过 4 轮迭代，输出从"零散的数据罗列"变成了"结构化的投资分析报告"。

**5. 输出结果**

4 位 AI 专家 + Orchestrator 调度中心，Prompt 工程经过 4 轮迭代。
