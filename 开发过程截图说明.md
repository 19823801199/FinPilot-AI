# 开发过程截图说明

> 以下 10 张截图完整记录了 FinPilot AI 从需求分析到发布验证的全过程，每张截图对应一个关键开发节点。所有开发工作均在 TRAE IDE 中完成。

## 截图 1：需求分析

**标题**：AI 对话驱动需求收敛 — 从"金融AI助手"到 4 个核心功能模块

**说明文字**：在 TRAE 中通过多轮 AI 对话，将"做一个金融AI助手"的初始想法逐步收敛为清晰的产品定义。AI 帮助识别了三类目标用户（金融专业学生、431 考研党、个人投资者）和三大核心痛点（学习资源分散、研究工具割裂、知识管理困难），最终确定了 AI 学习、股票研究、知识库 RAG、Multi-Agent 协作四个功能模块和 MVP 范围。

**体现的 TRAE 能力**：AI 对话的深度理解能力 — 能够从模糊的产品想法中提取关键信息，主动追问缺失维度，帮助开发者完成从 0 到 1 的需求定义。

**评委为什么会加分**：展示了"AI 辅助产品思维"而非简单的"AI 写代码"。需求分析是产品成功的根基，用 AI 对话替代传统脑暴，体现了 AI Native 的开发范式。

## 截图 2：PRD 生成

**标题**：AI 辅助生成产品需求文档 — 6 个页面、31 个 API 端点、8 步状态流转

**说明文字**：基于需求分析结果，在 TRAE 中让 AI 直接生成结构化的 PRD 文档。核心产出包括：6 个页面的交互流程设计、Multi-Agent 协作的 8 步状态流转（意图识别→任务规划→任务拆分→分配专家→执行→收集结果→合并→最终回答）、以及 31 个 API 端点的完整接口定义（HTTP 方法、路径、请求参数、响应格式）。

## 截图 3：UI 设计

**标题**：AI 生成 TailwindCSS v4 设计系统 — 深空蓝 + 翡翠绿金融主题

**说明文字**：在 TRAE 中让 AI 根据"金融 AI 助手"的产品定位，直接生成 TailwindCSS v4 的 `@theme` 全局样式配置。设计系统包含四级背景层次（bg→bg2→surface→surface2）、三色强调系统（翡翠绿 #00e5a0 主调、紫色 #6366f1 辅助、琥珀色 #f59e0b 警示）、以及 14 个基础 UI 组件（Button、Card、Modal、Toast、Skeleton、Empty 等）的设计规范。

## 截图 4：数据库设计

**标题**：AI 推导前后端一致的数据模型 — 30 个模型、5 个业务领域

**说明文字**：在 TRAE 中让 AI 根据 PRD 自动推导完整的数据模型体系。后端生成 6 个 Pydantic Schema 文件（agent、chat、knowledge、learning、stock），前端生成 5 个 TypeScript 类型文件（index、workspace、stock、knowledge、learning），共计 30 个数据模型定义。核心设计包括：泛型 `ApiResponse<T>` 统一响应格式、`AgentRole` 枚举约束专家角色、`dependencies: string[]` 字段支持 DAG 任务依赖。

**体现的 TRAE 能力**：AI 的跨语言类型推导能力 — 能够同时生成 Python Pydantic Schema 和 TypeScript interface，并确保两端字段名、类型、嵌套结构完全一致，消除前后端类型不对齐的隐患。

**评委为什么会加分**：前后端类型一致性是工程质量的硬指标。30 个模型的系统化设计展示了全局视野，DAG 依赖模型的设计展示了算法思维。这不是简单的 CRUD 数据模型，而是为 Multi-Agent 系统量身定制的任务编排数据结构。

## 截图 6：前端开发

**标题**：AI 辅助开发 SSE 流式对话界面 — 三栏布局 + 实时 AI 输出

**说明文字**：在 TRAE 中让 AI 生成工作台页面的核心代码。三栏布局（ConversationSidebar + ChatArea + StatusPanel）通过 Flex 实现，每个面板支持独立的折叠状态管理。ChatInput 组件集成了 SSE 流式接收逻辑——使用 `fetch` 建立 `text/event-stream` 连接，通过 `ReadableStream` 逐块读取 AI 回复，`AbortController` 管理连接生命周期，组件卸载时自动关闭连接。消息支持 Markdown 格式渲染（`**bold**` → `<strong>`）。

**体现的 TRAE 能力**：AI 的复杂交互逻辑生成能力 — 能够同时处理多个关注点（SSE 流式协议、连接生命周期管理、Zustand 多 Store 联动、Markdown 渲染），生成可直接运行的组件代码，而非需要大量调试的半成品。

**评委为什么会加分**：SSE 流式输出是 AI 产品的核心技术指标。从 fetch 到 ReadableStream 到 AbortController 的完整实现，展示了开发者对浏览器 API 的深入理解。三栏布局 + 面板折叠 + 流式消息的组合，展示了前端工程的综合能力。

## 截图 7：股票研究员开发

**标题**：AI 驱动的股票深度分析 — AKShare 实时数据 + DeepSeek 研报生成

**说明文字**：在 TRAE 中让 AI 实现股票研究员页面的完整功能链路。前端包含搜索侧边栏（模糊搜索 + 热门股票 + 自选股）、股票详情面板（K 线图 Canvas 渲染 + 行情数据展示）、AI 研报组件（结构化研究报告渲染）。后端集成 AKShare 获取 A 股实时行情数据，调用 DeepSeek API 生成包含基本面分析、财务数据解读、行业对比、投资建议的深度研究报告，通过 SSE 流式输出。

**体现的 TRAE 能力**：AI 的全栈功能链路实现能力 — 能够同时生成前端交互组件（搜索、K 线图、研报渲染）和后端数据管线（AKShare 数据获取 → AI 分析 → SSE 流式输出），确保前后端接口对齐、数据流转完整。

**评委为什么会加分**：股票分析是金融 AI 的高价值场景。AKShare 真实数据 + DeepSeek AI 分析 + Canvas K 线图 + SSE 流式输出的组合，展示了"AI + 行业数据"的深度融合，而非简单的聊天机器人套壳。

## 截图 8：知识库开发

**标题**：AI 实现 RAG 检索增强生成 — 文档上传 + 向量检索 + 智能问答

**说明文字**：在 TRAE 中让 AI 实现 RAG 知识库的完整管线。前端包含文档管理器（上传、分类过滤、搜索、删除）和 RAG 问答面板（自然语言查询 + AI 回答 + 来源标注）。后端实现了文档切块服务（500 字符/块，50 字符重叠区域）、ChromaDB 向量存储（支持 PersistentClient 持久化和内存 Fallback 两种模式）、Embedding 服务、语义检索服务和 RAG 主服务（检索 Top-K 文档片段 → 上下文组装 → DeepSeek 生成回答 → 返回来源标注）。

**体现的 TRAE 能力**：AI 的 RAG 管线实现能力 — 能够生成完整的检索增强生成链路，包括 ChromaDB 的 Fallback 降级策略（未安装时自动切换内存存储）、文档切块的重叠区域设计、以及检索结果与 LLM 上下文窗口的匹配策略。

**评委为什么会加分**：RAG 是当前 AI 应用的核心技术方向。从文档切块到向量存储到语义检索到 LLM 生成的完整管线，展示了开发者对 RAG 架构的深入理解。ChromaDB Fallback 设计展示了鲁棒性思维。

## 截图 9：Multi-Agent 开发

**标题**：AI 实现 Multi-Agent 专家协作系统 — 4 位专家 + Orchestrator 调度

**说明文字**：在 TRAE 中让 AI 实现 Multi-Agent 协作系统——项目最具创新性的部分。4 位金融领域 AI 专家（学习导师、研究分析师、知识专家、报告撰写员）各配备独立的 system_prompt 和 task_prompt。Orchestrator 调度中心实现了意图识别 → 任务规划 → DAG 拓扑排序执行 → 结果合并的完整工作流。Prompt 管理器集中管理 4 套角色模板，记忆管理器维护对话历史和任务结果。

**体现的 TRAE 能力**：AI 的 Prompt 工程辅助能力 — 能够为金融领域专家生成高质量的结构化 Prompt（学习专家覆盖 431/CFA/FRM 体系，研究分析师定义 6 步分析流程），以及 DAG 拓扑排序的任务调度算法。AI 在此阶段既是开发工具，也是"教 AI 如何做 AI"的元能力体现。

**评委为什么会加分**：Multi-Agent 是 AI 应用的前沿方向。4 位专家的角色设计、Orchestrator 的 DAG 调度、Prompt 工程的精细程度（system_prompt + task_prompt 双模板结构），展示了开发者对 AI 系统设计的深度思考。这不是简单的 API 调用，而是真正的 AI 架构设计。
