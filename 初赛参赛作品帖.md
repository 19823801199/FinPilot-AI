## 一、Demo简介

### 项目背景

金融领域的知识获取一直存在三大痛点：信息过载、工具割裂、学习低效。金融专业学生备考 431 金融学综合时，面对数百页教材和零散的真题，缺乏系统化的学习路径；个人投资者想分析一只股票，需要在行情软件、研报平台、AI 工具之间反复切换；研究人员管理大量 PDF 文档，却无法快速检索和问答。FinPilot AI 正是为解决这些真实痛点而诞生的 AI 金融智能工作台。

### 核心价值

FinPilot AI 将 AI 金融学习、AI 股票研究、AI 知识库管理三大核心能力整合在一个统一的工作台中，并通过 Multi-Agent 专家协作系统，让不同领域的 AI 专家像人类团队一样协同工作。用户无需在多个工具之间切换，一个平台即可完成从知识学习到投资分析的全流程。

### 目标用户

| 用户群体 | 核心需求 | 使用场景 |
|---------|---------|---------|
| 金融专业学生 | 系统化学习 431 金融学综合考点 | 知识点讲解、配套练习、错题追踪 |
| 431 金融专硕考研党 | 高效备考、查漏补缺 | 章节学习、真题练习、学习统计 |
| 个人投资者 | 快速获取股票分析、研报解读 | 股票搜索、AI 研报、自选股管理 |

### 核心功能介绍

**1. AI 金融学习助手**

覆盖 431 金融学综合全部考点，提供系统化的课程体系。每门课程包含知识点讲解、配套练习题、错题本管理。AI 根据用户答题情况动态调整学习路径，薄弱知识点自动强化。

**2. AI 股票研究员**

集成 AKShare 真实 A 股行情数据，支持股票代码/名称模糊搜索、实时行情展示、K 线图分析。AI 基于 DeepSeek 大模型生成深度研究报告，涵盖基本面分析、技术面解读、投资建议。自选股管理功能支持价格变动追踪。

**3. AI 知识库 RAG**

支持 PDF、Markdown 等格式文档上传，通过 ChromaDB 向量数据库进行语义存储。用户可通过自然语言查询知识库内容，AI 基于检索到的相关文档片段生成精准回答，并标注信息来源。

**4. Multi-Agent 专家协作**

四位 AI 专家协同工作：学习专家负责知识讲解，研究分析师负责股票分析，知识专家负责文档问答，报告撰写专家负责结果整合。Orchestrator 调度中心自动识别用户意图、规划任务、分配专家、合并结果。

### 使用场景

**场景一：考研备考**

小李正在备考 431 金融学综合。他打开 FinPilot AI 的学习中心，选择"货币银行学"课程，AI 学习专家为他讲解货币层次理论。讲解完毕后，系统自动生成 5 道配套练习题。小李答错 2 题，错题自动收录到错题本，AI 针对薄弱知识点再次讲解。

**场景二：股票分析**

老王想分析贵州茅台的投资价值。他在工作台输入"帮我分析一下贵州茅台"，Orchestrator 识别为股票分析意图，自动调度研究分析师和报告撰写专家。研究专家调取 AKShare 实时数据，生成包含 K 线图、基本面数据、行业对比的深度研报，报告撰写专家将结果整理为结构化的投资建议。

**场景三：研报问答**

研究员小张上传了 10 份新能源行业研报到知识库。他输入"总结一下新能源行业的主要投资机会"，知识专家从向量数据库中检索相关文档片段，AI 基于检索结果生成带来源标注的综合回答。

## 二、创作思路

### 灵感来源

灵感来自三个真实场景。第一个场景是备考 431 金融学综合时，教材内容庞杂，缺乏针对性的练习和反馈。第二个场景是日常投资中，想看一只股票的分析报告，需要在多个平台间切换，耗时费力。第三个场景是阅读大量研报后，想快速找到某个观点的出处，却要从几十份文档中逐一查找。这三个场景的共同点是：信息分散、工具割裂、AI 能力未被充分利用。

### 用户痛点

**痛点一：学习资源分散**

431 金融学综合涵盖货币银行学、公司理财、投资学三门课程，知识点超过 200 个。考生需要在教材、网课、真题之间反复切换，缺乏一体化的学习平台。

**痛点二：研究工具割裂**

投资者分析一只股票，需要打开行情软件看 K 线、打开研报平台看分析、打开 AI 工具做总结。三个工具之间的信息无法打通，分析效率低下。

**痛点三：知识管理困难**

研究人员积累的 PDF 研报、笔记文档越来越多，传统文件夹管理方式无法支持语义检索。想找到某个观点的出处，往往需要翻阅数十份文档。

### 为什么选择这个方向

金融是 AI 落地的高价值场景。一方面，金融知识具有专业性强、时效性高的特点，通用 AI 难以给出精准回答；另一方面，金融领域的用户付费意愿强，商业化路径清晰。更重要的是，这个方向将 AI 的三种核心能力（对话、检索、分析）与金融的真实需求深度结合，而非简单的"套壳"应用。

### 为什么选择 AI

没有 AI，FinPilot 只能是一个信息聚合平台。AI 的加入带来了三个质变：第一，AI 学习助手可以根据用户答题情况动态调整学习路径，实现个性化教学；第二，AI 股票研究员可以基于实时数据生成深度分析报告，替代人工研究的大量重复工作；第三，RAG 知识库让私有文档具备了语义理解能力，这是传统搜索无法实现的。

### 为什么选择 TRAE

TRAE 的 AI 辅助编程能力大幅提升了开发效率。在开发过程中，TRAE 帮助完成了代码生成、Bug 修复、架构设计、文档编写等工作。特别是在 Multi-Agent 系统的开发中，TRAE 协助设计了 Orchestrator 调度逻辑和 Agent 间的通信协议。TRAE 的上下文理解能力让跨文件重构变得高效，这是传统 IDE 无法比拟的。

采用 Monorepo 架构，前端 Next.js 15 + 后端 FastAPI。TRAE 协助完成了项目初始化、目录结构规划、环境变量配置、Docker 配置文件编写。

TRAE 的代码补全和重构功能大幅提升了组件开发效率。

### 后端开发

在 TRAE 中完成全部后端服务开发：

- **API 路由层**：6 个路由模块，31 个端点，统一 `/api/v1` 前缀
- **业务服务层**：20 个服务类，涵盖聊天、股票、知识库、学习、Agent 等领域
- **数据模型层**：Pydantic Schema 定义，类型安全
- **核心模块**：AI 客户端（DeepSeek）、缓存、配置、日志

### RAG开发

RAG 系统的开发是项目的技术亮点之一。在 TRAE 的辅助下，完成了：

- **文档切块服务**：基于字符数的智能文本分割，支持重叠区域
- **Embedding 服务**：调用 DeepSeek API 生成文本向量
- **ChromaDB 集成**：向量存储、语义检索、元数据过滤
- **RAG 管线**：检索 -> 上下文组装 -> LLM 生成 -> 来源标注

### Multi-Agent开发

Multi-Agent 系统是项目最具创新性的部分。在 TRAE 的辅助下，实现了：

- **Agent 管理器**：4 位专家的注册、状态管理、任务分配
- **调度中心（Orchestrator）**：意图识别 -> 任务规划 -> 专家分配 -> 结果合并
- **工作流管理器**：任务依赖追踪、执行状态监控
- **记忆管理器**：对话历史、任务结果持久化
- **Prompt 管理器**：集中化管理各专家的系统提示词

### Bug修复

开发过程中遇到的主要问题及修复：

- **XSS 漏洞**：chat-area 组件使用 dangerouslySetInnerHTML 渲染 AI 输出，未做 HTML 转义。修复方案：增加 escapeHtml 函数，对 LLM 输出进行转义后再渲染。
- **流式输出中断**：SSE 连接在组件卸载时未正确关闭。修复方案：使用 AbortController 管理连接生命周期。
- **股票搜索性能**：AKShare 搜索接口响应慢。修复方案：增加内存缓存层，热门股票预加载。

## 五、踩坑与优化

### 开发过程中遇到的问题

**问题一：ChromaDB 在 Windows 环境安装失败**

ChromaDB 依赖的 onnxruntime 在部分 Windows 环境缺少 VC++ 运行时库，导致 pip install 报错。

**解决方案**：在 requirements.txt 中将 chromadb 标记为可选依赖，提供预构建的 Docker 镜像作为替代方案。开发环境使用 SQLite 模式运行，生产环境使用 Docker 部署。

**问题二：SSE 流式输出与 Next.js 的兼容性**

Next.js 的 fetch 在开发模式下会触发多次请求，导致 SSE 连接重复建立。

**解决方案**：使用原生 EventSource API 替代 fetch，手动管理连接状态。在组件卸载时调用 AbortController.abort() 关闭连接。

**问题三：Multi-Agent 任务依赖执行顺序**

早期实现中，Agent 任务按固定顺序执行，未考虑任务间的依赖关系。例如报告撰写专家需要在研究分析师完成后才能开始工作。

**解决方案**：引入 DAG（有向无环图）调度算法，每个任务声明依赖列表，调度器按拓扑排序执行。实现代码如下：

```python
while pending:
    ready = [t for t in pending.values()
             if all(dep in executed for dep in t.dependencies)]
    for task in ready:
        await self._execute_task(task)
        executed[task.id] = task
        del pending[task.id]
```

**问题四：RAG 检索结果相关性不足**

初期使用简单的关键词匹配，检索结果与用户问题的相关性较低。

**解决方案**：引入 Embedding 语义检索，将文档和查询都转换为向量，通过余弦相似度计算语义相关性。同时增加重排序（Rerank）步骤，提升 Top-K 结果的准确性。

### 关键优化

**优化一：Zustand Store 性能**

初期多个组件解构整个 store，导致不必要的重渲染。优化后使用选择器模式，每个组件只订阅需要的状态片段。

**优化二：股票数据缓存**

AKShare 接口调用频繁且响应慢。增加内存缓存层，热门股票数据缓存 5 分钟，减少重复请求。

**优化三：文档切块策略**

初期按固定字符数切块，导致句子被截断。优化为按段落边界切块，保持语义完整性，块间保留 50 字符重叠区域。
